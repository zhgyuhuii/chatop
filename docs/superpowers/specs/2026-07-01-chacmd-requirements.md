## 1. 项目背景与目标

### 1.1 愿景
在 chatop-ai（单 KasmVNC 智能体桌面容器）之上，建设一个**现代化的智能体综合指挥调度平台**：把任务智能分配到每个容器智能体，指挥系统自身也集"智能体 + 知识库 + 指挥调度"于一体，并横向指挥调度其它容器。**核心目标就一句话：指挥调度智能体。**

### 1.2 定位与护城河（已确认）
- **定位**：统一指挥、**可操作整机桌面的异构 agent 舰队**——每个 agent 拥有一整台 OS 桌面容器（真 GUI 应用、可 computer-use）+ 免密远程桌面 + 异构 CLI agent（OpenHands/Claude Code/Codex/Hermes/OpenClaw）作一等公民。
- **护城河**：上述"能操作整机桌面的 agent 舰队"是通用编排产品（Dify/Coze/百炼/LangGraph）不具备的差异点。ChaCMD **不是"又一个多 agent 编排器"**。

### 1.3 Beachhead 与商业模式（已确认）
- **Beachhead 场景 = S1 软件开发流水线**（开发 + 沙箱跑 + 桌面预览）。选它因为**单个聊天 agent 明显做不到**（需真运行时 + GUI 预览 + 沙箱），MVP 能证明"多容器 agent 舰队 > 单个强 agent"的价值，而不只是证明管道。
- **商业模式 = 私有化 License（租户 = 部门，成本 = 部门分摊）**。契合国内/离线/RBAC/数据不出境侧重。**不做 SaaS 用量计费**；多租户隔离务实降级（部门级 RLS + per-job 卷隔离即可）。

### 1.4 三种部署/打包形态（见设计 §3.9）
平台以**一套代码 + 配置切换**支持三形态；ChaCMD 后台可配置"察元接入模式"：
- **形态 A · 察元纯净**（现状）：不启动指挥系统，界面即现在察元，打包只含现有功能。
- **形态 B · ChaCMD 全量**：察元全量 + 指挥系统，主页为指挥大屏，打包全量（内嵌察元）。
- **形态 C · ChaCMD 轻量挂载**：**不打包察元**，全 HTTP 对接**已独立部署的察元**（接口 + 界面直接跳转/iframe），极轻。`P1`；前提=补齐察元 HTTP 接口 + 跨实例 SSO 免登（简单版一次性 token / 标准版 OIDC）。
> 关键实现约束：ChaCMD 后端一律经 `ChayuanClient` **全 HTTP** 抽象对接察元（B=localhost，C=远程），B/C 后端同一套代码；前端 `chayuanUiMode = embedded | mounted` 切换察元界面来源。**P0 即按此抽象写**（默认 localhost），使 C 只需换 URL + 补 SSO，不返工。

### 1.5 与母体察元的关系（#17）
- **以察元 `/work/chayuan-desktop` 为母体内核**：复用其模型网关（25+ provider + 本地）、RBAC/组织/审计、知识库 kb-query、agent 框架。
- **ChaCMD 只补"跨容器横向调度"增量**。母体做规划/知识/网关/RBAC（**非实时热路径**）；**实时派活核心独立为薄层**，不经察元重 monorepo（避免重母体拖累低延迟）。
- 治理：monorepo 双 SKU + 单向依赖 + feature-flag + 防腐层；察元仍可独立部署、与上游同步、低耦合。
- **合规约束**：察元 AGPL-3.0，合并 SKU 即 AGPL；对外分发/SaaS/嵌第三方 AGPL 会触发源码义务（须显式决策）。

## 3. 用户角色与视图（多用户 / 多角色）

平台为**两套壳、三种视图**，由 RBAC（#14）+ BrandConfig 决定落地视图：

| 视图 | 使用者 | 看到 / 能做 | 提供方 |
|---|---|---|---|
| **指挥大屏** | 管理员 / 指挥官 | 全局态势：所有主机/容器/任务/成本/编队；直接下令 | 指挥中心 |
| **员工视图** | 普通员工（S7 云桌面）/ agent 自省 / 非管理员默认 | **自己这一个容器**：我的状态 / 我的成果 / 我的所有任务工作记录；操作自己的桌面 | **容器常驻服务（agent-bridge）** |
| **访客视图** | 访客 | 被分享的单个任务结果（只读） | 指挥中心（分享链接） |

**员工视图必须能看**（用户明确）：①我的状态（在线/忙闲/当前任务/连的模型/角色契约）②我的成果（产物/预览/交付物，可下载）③所有任务工作记录（履历时间线：目标/步骤日志/事件回放/耗时/token/A2A 交接/审批）。工作记录**真源在中心**（容器可重启/销毁），实时态读本地。

## 4. 功能需求（21 条，按功能域组织）

> 每条：**描述** · 验收要点 · 优先级（P0/P1/P2/P3）· 追溯设计章节。原始编号 #1–#20 保留用于追溯。

### 4.1 容器与集群编排
- **#2 复制/供给容器**：复制本容器并配端口/文件/用户名/密码/权限/动态装应用/启动。`验收`：一键从基座镜像起一个可用 agent 容器。`P0` · §6.3
- **#3 容器命名/昵称**：注册表 `nickname → container` 映射，供喊昵称派活。`验收`：昵称唯一、可解析到容器。`P0` · §6.3
- **#13 容灾/高可用**：某角色容器挂了立即起相同职责替身；心跳探活 + 期望态协调环重建/改派 + 在途任务重放。`验收`：杀掉容器后自动重建并续跑（P0 单机、P2 跨机）。`P1→P2` · §6.9
- **部署纳管与编排**：主机接入 → 自动部署基础容器 + 智能体容器 → 容器与中心通讯打通 → 中心检测在线 → 中心下发装应用/技能。两层 bridge（host-bridge 管主机 Docker / agent-bridge 管容器执行）。`验收`：新主机 bootstrap 后中心可见并可下发部署。`P1` · §6.19
- **子容器内置能力**：本容器作为 agent 工作站必须内置 agent-bridge / agent 运行时 / 工作目录 / 事件流适配 / 模型网关对接 / 员工视图服务等（清单见 §6.18）。`P0 起` · §6.18

### 4.2 调度与编排
- **#5 统一调度对话框**：说事 / 喊昵称 → 路由到对应容器 agent 干活。`验收`：@昵称直派命中；NL 路由（P1）。`P0`（直派）· §6.1
- **#11 多智能体串并行编排**：任务可串可并，agent 间可串可并；支持有界回环 + 条件路由 + 子编排（sub-DAG）。`验收`：一个 DAG 含串行链 + 并行扇出/汇聚 + 审核回环可执行。`P0`（线性）→`P1`（复杂 DAG）· §6.7
- **#6 实时反馈**：各容器 agent 执行任务实时流式回传指挥系统。`验收`：agent 输出 JSONL/stream-json 实时到大屏。`P0` · §6.6
- **#7 结果聚合 + 成果共享**：结果回对话框 + 容器间数据/成果物/参考文件共享；中心可汇总。`验收`：产物落卷、中心语义汇总回大屏。`P0` · §6.4

### 4.3 角色与权限
- **#1 全套角色分配**：给每个 agent 定角色（能力画像 + system prompt + 绑定技能）。`验收`：可给容器/agent 指派角色。`P0` · §6.13
- **#18 角色契约（Charter）**：声明式定义身份/目标/系统提示词/技能/触发(trigger)/交接(handoff)/知识范围；派活时注入各 agent。`验收`：数据来了 agent 知道做啥、知道下一步传给谁。`P0`（system_prompt 注入）→`P1`（触发/交接）· §6.13
- **#14 资源级 RBAC**：主体(用户|角色) × 资源(容器/系统视图) × 动作(查看/开界面/派活/配)。私有化下租户=部门。`验收`：无权的资源不出现；越权被拒。`P0`（部门 RLS）→`P1` · §6.10
- **#9 定/调角色 + 下发安装**：统一装应用/agent/skill + 定/调角色（参考 Hermes 固定角色）。`验收`：中心可下发安装与角色配置。`P1` · §6.20

### 4.4 知识与数据
- **#19 知识源查询范围授权**：给智能体分配可查知识库范围（文档类/数据库类/向量库类等）+ 查询权限。主体 × 知识源(类型粒度 + 具体库粒度) × 动作。`验收`：检索按范围过滤，越界库不可见。`P1` · §6.14
- **#15 外部数据接入**：某容器可拉数据(轮询)/接数据(webhook/MQ)/感知外部数据(事件触发编排)；信任分级。`验收`：外部事件可归一化落卷并触发编排（默认不直接驱动高权限 agent）。`P3` · §6.11
- **#4 统一工作空间**：同一任务同一文件夹——容器内固定 `$HOME/<code>` 挂共享卷 `/workspace/<job_id>/`。`验收`：任务产物统一落该目录。`P0`（per-job 卷隔离）· §6.4

### 4.5 模型与配置
- **#10 统一配置中心**：统一设置每个 agent 的模型/API Key/Base URL 等。`验收`：中心配置下发到容器 env。`P0`（基础注入）· §6.2
- **#16 统一模型网关**：agent 与模型间插统一网关做协议转换 + 路由 + 凭据收口；接国内/本地模型（DeepSeek/Qwen/Ollama/vLLM），离线可跑；为 Claude Code/Codex 补 Anthropic `/v1/messages` shim。`验收`：agent 经网关调国内/本地模型成功；Key 不落容器明文。`P0` · §6.12

### 4.6 智能体协作
- **#12 A2A 直连**：A 的结果不经指挥系统直传 B——数据走共享卷直传 + 消息走总线 pub/sub，指挥台旁路审计；防环 hop/TTL。`验收`：两 agent 直传数据、中心可审计。`P3` · §6.8
- **容器通讯无 IP 寻址**：容器间及容器↔中心通讯**一律用逻辑标识**（昵称 #3 / job_id / 总线 subject / 共享卷路径），**不靠 IP、任何模块不得硬编码容器 IP**；容器重建/迁移/替身接管后寻址自动重解析。`验收`：全链路寻址无容器 IP 硬编码。`P0`（无 IP 原则）→ 点对点覆盖网直连 `P1+` · §6.22 / I10

### 4.7 界面与交互
- **#8 免密打开容器界面**：指挥系统免密打开任意容器 OS 界面（凭据金库 + 一次性 token 预认证反代）；区分只读/可操作。`验收`：大屏点容器一次性 token 免密开 KasmVNC，越权不可开。`P2` · §6.5
- **指挥大屏（管理员）**：全幅态势画布 + 悬浮指挥条 + 可折叠抽屉 + 事件流分层；投墙模式 = 画布减命令条。`验收`：一屏统揽态势并可下令。`P0`（最小）→`P1`（完整）· §6.15
- **员工视图（容器常驻）**：见 §3。`验收`：员工登录默认落员工视图，看到状态/成果/工作记录。`P1` · §10.3 / §6.18-G4
- **新建任务**：任务名称 + 任务代号 + 工作目录 + 智能体选择/创建；仅 goal 必填，其余默认；NL 草稿原地渲染成带"推断"标记的表单；"创建智能体"与"派活"分离。`验收`：填 goal + 选 agent 即可建任务。`P0`（最小）→`P1` · §6.16

### 4.8 应用与技能
- **技能与插件体系**：装通用 **SKILL.md**（agentskills.io 标准，跨 OpenClaw/Hermes/Claude Code/OpenHands 通用，一次装多 agent 共用）+ MCP 插件；来源 ClawHub / Hermes Skills Hub / 察元 `/skills` API；chatop-ai 应用管理窗口详情页可进技能安装页搜索/装。技能可绑角色契约。`验收`：中央搜索→下发→绑角色；容器内 agent 共用。`P1`（生产禁自动装，需 allowlist+签名+沙箱）· §6.20

### 4.9 对外集成
- **#20 Task-as-API（任务代号即对外调用契约）**：每个任务有 `code`，**外部凭 code 即可调用**中心或智能体接口——执行任务(invoke) / 对接数据(拉·接) / 提交数据(同步·webhook·回写)。`code` = 对外契约 + 任务模板标识（可复用），传 code → spawn 新实例。per-caller API Key（部门级）+ RBAC + 幂等键 + 自动 OpenAPI；外部输入按信任分级降权；webhook HMAC 签名。默认走中心，智能体直连经统一网关（不裸暴露端口）。与 #15 互补（主动调用 vs 被动感知）。`验收`：外部传 code + inputs 触发任务，取回结果或收到 webhook 回调。`P1`（P0 先存好 code/schema/幂等位）· §6.21

### 4.10 基础设施与服务治理
- **#21 服务注册发现 + 配置中心可插拔（Nacos 可选接入）**：万级多实例（host-bridge/agent-bridge/Orchestrator/gateway）需服务注册发现 + 统一配置动态下发/热更新。设计为**可插拔 provider**：`ServiceRegistry`/`ConfigSource` SPI——**内置实现（自研/Postgres）为默认；可选接入 Nacos（服务发现 + 配置中心，国产友好）；预留 Consul/etcd/Eureka**。**ChaCMD 后台可配置"服务发现/配置源 = 内置 | Nacos | …"，开关接入或不接入**（小规模零依赖，大规模/已有 Nacos 的企业接入）。配置项：Nacos 地址/命名空间(按部门)/分组/鉴权/开关。`验收`：后台切换到 Nacos 后各实例经 Nacos 注册发现 + 拉配置；切回内置仍可运行。`P1`（P0 先按 SPI 抽象写内置实现，不锁死）· §10.4-B9
  - **子项 · Spring Cloud/OpenFeign 互操作**：ChaCMD（Python FastAPI）**须能被 Java OpenFeign 按服务名调用**（OpenFeign 是 Java 客户端、服务端语言无关）。三前提=标准 REST（天然）+ 注册 Nacos（`nacos-serving-python`/`nacos-sdk-python` v2）+ OpenAPI 契约（#20）。反向 Python 调 Java 用 nacos 发现 + httpx（或 uplink）。异构靠 Nacos+REST+OpenAPI，不用 Java 重写核心。`验收`：Java Feign 按服务名调通 ChaCMD 接口。`P0`（REST+OpenAPI）→`P1`（Nacos 发现）· §6.23

## 6. 非功能需求（NFR）

> 由四视角（架构 / 产品 / 设计UE / 平台性能安全）深度评审得出，多为四方共识。详见设计文档 §10.1。

### 6.1 性能 / 高并发 / 低延迟
- **NFR-P1 常驻会话执行**：派活适配器支持"one-shot 批量"+"persistent-session 流式"两模式；实时/长时任务用常驻会话（禁 subprocess-per-task）。`P0`
- **NFR-P2 热路径禁 LLM + 双路径**：实时决策路径走确定性规则 + 缓存结构化数据（`synthesize=false`），LLM 仅用于冷启 NL 路由/离线策略；派活分快慢路（@昵称/规则/缓存=快）。`P0→P1`
- **NFR-P3 事件总线**：主总线 NATS JetStream（NAT 友好 + 多租户 subject 隔离 + 载 traceparent）；Redis Streams 仅 MVP，Kafka 作外部数据/审计二级 sink。**心跳与"判死"解耦**（独立 lease，禁与业务流混一条总线）。`P0`
- **NFR-P4 连接管理**：海量反连长连接需分片 + 背压（credit flow control）+ 事件合并/降采样 + 可恢复会话（cursor 续传）+ 抖动退避防惊群。`P1`
- **NFR-P5 延迟预算**：定义派活/冷启/首 token/端到端延迟 SLO 并度量。`P1`

### 6.2 可扩展性
- **NFR-S1 Orchestrator 无状态化**：状态外置 Postgres（弃 SQLite）；连接层拆独立无状态 bridge-gateway；按 job_id 分区 active-active。`P0`（外置）→`P2`（多活）
- **NFR-S2 reconcile 单活**：容灾协调环必须 leader election/lease（防多实例双拉替身）。`P1`
- **NFR-S3 多机调度**：跨节点 placement/health/reconcile 先评估 Nomad，再决定是否自研（不重造 K8s）。`P1→P2`

### 6.3 多租户 / 多用户 / 多角色
- **NFR-T1 部门级隔离**：ChaCMD 新表全加 `tenant_id`（=部门）+ Postgres RLS；共享卷 **per-job 子目录 bind（禁整卷挂载）**；NATS Accounts 部门隔离（P1）。`P0`（RLS+卷）
- **NFR-T2 三视图**：管理员大屏 / 员工视图 / 访客，靠 RBAC + BrandConfig 分流。`P0`（数据）→`P1`（UI）
- **NFR-T3 部门配额**：网关 per-部门预算护栏（非 SaaS 计费）。`P1`

### 6.4 可靠性 / 一致性 / 容灾
- **NFR-R1 三者一致性**：Job 态(DB) / 产物(卷) / 容器实况 每事实单一真源；产物 write-then-atomic-rename + 完成标记；reconcile 兼收 Job 态（死容器 Task→interrupted）；承认至少一次 + 去重（不假设 code-exec agent 幂等）。`P0`
- **NFR-R2 耐久编排**：P0/P1 自研引擎锁死无耐久（崩溃=重跑整 Job）；P2 切 Temporal/Restate/DBOS 拿续跑/定时/幂等/human-signal。`P2`

### 6.5 模块化 / 可插拔 / 代码复用
- **NFR-M1 真插件 ABC**：ChaCMD 侧自建 `AgentAdapter`（spawn/stream/inject-charter/health/cancel + capability manifest）与 `Connector` ABC + 注册表，使加 agent/connector/skill **零改察元核心**。`P0`（按 ABC 写 OpenHands adapter）→`P1`
- **NFR-M2 技能可插拔**：SKILL.md 开放标准，一次装多 agent 共用。`P1`
- **NFR-M3 复用**：母体 libs 稳定部分直依赖（过内部 adapter + pin 版本）、业务能力走 HTTP+ACL；前端 packages 下沉 + 薄 app 壳 + BrandConfig 注入双 SKU。`P0`

### 6.6 跨平台
- **NFR-X1**：母体察元 Tauri 2 跨 Win/macOS/Linux/麒麟/UOS；后端 FastAPI 跨平台；子容器 Linux/KasmVNC。ChaCMD **以服务端/网络部署为主**（桌面单机 SKU 实质=纯察元，不硬撑本地指挥平台）。`P0`

### 6.7 可观测性
- **NFR-O1**：OpenTelemetry W3C traceparent 全链路（注入总线 header，job_id 作 baggage）；Prometheus metrics（重点 bus consumer lag / 派活延迟 / WS churn / reconcile / token 花费）；结构化日志（tenant/trace/job/task id）；SLO 告警。`P0`（最小 trace）→`P1`（全套）

### 6.8 安全
- **NFR-SEC1 沙箱**：跑不可信代码的容器用 **rootless Docker（P0）/ gVisor/Kata（P1）**；容器加固基线（cap-drop ALL / no-new-privileges / seccomp / userns-remap / read-only rootfs / cgroup 硬限）；**Docker socket 绝不入不可信容器**（禁 DinD-privileged / 裸 sibling-socket）。`P0`
- **NFR-SEC2 出站管控**：默认拒绝出站 + per-role 白名单（仅网关 + 批准端点）。`P0`
- **NFR-SEC3 技能供应链**：生产禁自动装；allowlist + 签名校验 + 沙箱执行 + 私有 registry + pin 版本。`P1`
- **NFR-SEC4 间接 prompt 注入**：外部数据/KB 进 prompt 后，执行必须在工具/网关层强制降权（不靠提示词自觉）。`P1`
- **NFR-SEC5 凭据金库**：密钥不落明文共享卷；虚拟 Key 短 TTL + 秒级吊销；生产用 Vault/SOPS。`P0`（不明文）→`P1`（Vault）
- **NFR-SEC6 审计完整性**：审计 append-only + 哈希链 + 独立 sink（WORM）。`P1`
- **NFR-SEC7 接入面**：外部/前端**永不直连容器**；网关端点 mTLS + 鉴权；一次性 token 绑租户 + 短有效期 + 审计。`P0`

### 6.9 成本治理
- **NFR-C1**：per-job token 硬预算（超则 kill）+ per-部门月预算 + per-role 上限 + 实时计量（接察元 `llm_callback`）+ runaway 熔断 + agent 最大迭代数；跑前预估 + 燃尽 + 部门分摊报表。护栏放网关层。`P0`（护栏）→`P1`（报表/预估）

### 6.10 人工介入（human-in-the-loop）
- **NFR-H1**：任务状态机含 `pending_approval` 一等状态；大屏"需处理"区带 [批准]/[打回]/[暂停]/[改派]/[回滚] 动作；#8 免密桌面作"接管"载体；步骤级介入（改产出再重跑）。`P0`（审批门）→`P1`（回滚/接管/步骤介入）

### 6.11 规模化与部署形态（万级 / 分布式 / 集群，见设计 §10.4）
> 以下按"目标是否为单体万级/跨地域"取舍（见 §10 岔路①②）。
- **NFR-SC1 默认无头轻容器**：默认无头轻量 agent 容器；重桌面容器（chatop-ai）按需起 + 用完回收（冷/热池、按需唤醒、空闲回收）。1 万 × 3GB 桌面常驻不可行。`P1`
- **NFR-SC2 连接层水平分片**：反连长连接终结层（bridge-gateway）独立成集群 + 一致性哈希路由 + 背压/续传；支撑万级持久连接。`P1`（万级目标下提前）
- **NFR-SC3 分层/联邦指挥（战区制）**：单体万级/跨地域时，全局中心 + 区域指挥节点 + 容器；区域自治 + 向上汇聚。非单体则"多个千级独立部署"即可。`P2`（按岔路②）
- **NFR-SC4 成熟编排与协调**：万级 + 多机房网络分区常态，底层容器编排/协调委托 K8s（信创场景=国产 K8s）/etcd，而非自研跨机 reconcile；覆盖网只管应用层 agent 调度。`P2`（按岔路②）
- **NFR-SC5 数据面扩展**：共享卷用分布式存储（Ceph/国产）分租户分区替单 NFS；状态库分库分表/读写分离或分布式库。`P1→P2`
- **NFR-SC6 大规模运维**：批量升级/灰度、准入控制、队列深度上限、多架构镜像流水线（x86+ARM+LoongArch）+ SBOM + 私有 registry。`P1→P2`

### 6.12 信创与等保 / 党政军合规（见设计 §10.4-A/B）
> 以下仅当**目标客户含党政军/涉密/国企信创**时为硬约束（见 §10 岔路①③）。P0 须预留接口，不必 P0 实现。
- **NFR-GX1 国产化栈适配**：CPU 鲲鹏/飞腾/龙芯、OS 麒麟/UOS、**DB 达梦/人大金仓/GaussDB（DB 访问层抽象，不锁 Postgres 方言）**、推理国产 NPU 昇腾 MindIE/寒武纪（非 CUDA）。`P0 预留接口 → P1/P2 实现`
- **NFR-GX2 国密算法**：SM2/SM3/SM4 + TLCP，密码学层可替换（不硬编码 TLS/AES/HMAC）。`P0 预留 → P1`
- **NFR-GX3 完全离线**：涉密网物理隔离，离线唯一（本地权重 + 国产 NPU）；对外调用限内网。`按岔路③`
- **NFR-GX4 三员分立 / SoD / 双人复核**：RBAC 扩展系统管理员/安全保密员/审计员权力制衡 + 双人操作复核。`P2`
- **NFR-GX5 内容安全 / 生成式 AI 合规**：agent 输出过内容安全网关；备案；涉政涉密过滤。`P2`
- **NFR-GX6 企业统一身份**：对接 LDAP/AD/OIDC/SSO + 国产身份（麒麟信安/宁盾）。`P1`

## 7. 约束与红线

- **R1 供应链**：**绝对禁用基于泄露源码的 Claude Code 分支**（claude-code-fork 等，法律侵权 + 投毒双禁）；用 OpenHands 或官方闭源 CLI 经 shim。
- **R2 依赖投毒**：LiteLLM PyPI `1.82.7/1.82.8` 为窃凭据投毒版，必须 pin 干净版 + 锁依赖。
- **R3 License**：1Panel GPL-3.0 已否决；产品不嵌第三方 copyleft。察元 AGPL-3.0 为自有产品可 relicense，但对外分发/SaaS/嵌第三方 AGPL 触发源码义务。
- **R4 沙箱**：Docker socket 不对外开 TCP、不入不可信容器；跨网 mTLS。
- **R5 授权**：无授权不访问 KB/source；授权判定只在服务端；前端/外部永不直连容器。
- **R6 查询**：查询阶段不接受外部 embedding model 参数（跟随知识库索引配置）。
- **R7 密钥**：不提交/不落明文 `.env`、密钥、token、dump、日志。
- **R8 信创/等保（若含党政军目标）**：不锁死 x86/Postgres/CUDA/标准密码——DB 访问层、密码学层、服务发现/配置源、模型后端**须从 P0 抽象为可替换接口**（不必 P0 实现国产，但不能硬编码到无法替换）。目标客户/规模/离线要求见 §9 岔路，未定前 P0 只做接口预留。

## 8. 优先级与分期

> ⚠️ **三个待确认岔路（决定架构边界，见设计 §10.4-D）**：①目标客户是否含**党政军/涉密/国企信创**？②规模形态是**单体万级/跨地域**还是**多个千级独立部署**？③是否要求**完全离线**？未定前，P0 只做**接口预留**（DB 访问层 / 密码学层 / 服务发现·配置源 SPI / 模型后端可换），不锁死 x86/Postgres/CUDA/标准密码，也不 P0 实现信创/联邦/K8s。

> P0 = 单 agent 明显做不到的**价值切片**（S1 开发 + 沙箱跑 + 桌面预览），并从第一天内建关键地基。详见设计文档 §10.2 / §10.3。

- **P0（价值切片 + 地基）**：#2 #3 #5(直派) #6 #7 #10(基础) #16 #1 #18(注入) #14(部门RLS) #4(卷隔离) #11(线性) + 指挥大屏最小 + 新建任务最小 + NFR-P1/P2/P3、S1/S2/T1/T2(数据)、R1/O1(最小)、SEC1/2/5/7、C1(护栏)、H1(审批门)。
  - **里程碑**：M1 骨架（Orchestrator+Postgres+NATS+bridge-gateway+反连注册）→ M2 派活+常驻执行+事件流 → M3 沙箱加固+iframe 预览+审批门+预算 kill+OTel → M4 端到端验收。
  - **P0 验收 5 条**：①端到端跑通建任务→沙箱开发+跑→预览→审批→汇总；②沙箱不可逃逸（socket 不入容器）；③超 per-job 预算自动 kill；④审批门可批准/打回；⑤大屏实时事件流 + 预览。
- **P1**：#13(单机容灾) #9 #19 部署纳管 员工视图UI 技能体系 #20 Task-as-API 任务模板库 通知 交付面 + NFR-S2/T1(Accounts)/T3/M1/M2/O1(全)/SEC3/4/6/C1(报表)/H1(回滚接管) + 大屏/事件流/DAG 完整。
- **P2**：#8 完整 跨机容灾 Temporal(NFR-R2) active-active(NFR-S1) Nomad(NFR-S3) 角色/agent 市场 合规呈现 S4/S5/S6。
- **P3**：#12 A2A #15 外部数据接入 投墙 S7 学习进化 + 复杂编排增强。
